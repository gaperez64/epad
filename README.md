# Tools for the existential fragment of Presburger arithmetic with divisibility
The first goal of this repository is to have a library to manipulate systems
of divisibility constraints over linear (diophantine) polynomials conjoined
with equations and inequalities.
1. Normalization of such formulas (based on ["On the Complexity of Linear
   Arithmetic with Divisibility", by Joël Ouaknine, Antonia Lechner and Ben
   Worrell.](https://www.cs.ox.ac.uk/people/james.worrell/LICS-main.pdf))
2. Solvers for normalized formulas (based on the same work mentioned above,
   and an alternative one based on ["Integer Programming with GCD
   Constraints", by Rémy Défossez, Christoph Haase, Alessio Mansutti, and Guillermo
   A. Pérez.](https://epubs.siam.org/doi/10.1137/1.9781611977912.128))

## Lipshitz' normalization

### High-level overview

The entry point is `LinDivs.norm` in `systems/lindivs.py`. It maintains a
worklist of candidate sub-systems. For each sub-system it performs the
following steps, branching disjunctively whenever a choice has to be made:

1. **Make left-hand sides non-negative.** For every divisor with some
   negative coefficient, enumerate the sign: either keep it as is, or flip it
   and add the implied inequality. This yields one or more sub-systems whose
   divisor LHS are non-negative (`all_disj_left_pos`).
2. **Eliminate equations and inequalities.** Use a decision procedure for
   linear integer arithmetic to obtain a (hybrid) semilinear representation
   of the solution set of the linear part, and substitute it back via an
   affine change of variables that does not introduce new variables. The
   result is a pure system of divisibilities (`all_disj_just_divs`, which
   delegates to `LinIneqs.solutions`).
3. **Reduce each divisibility by the gcd** of its coefficients, factoring
   out `-1` on the LHS when possible (`reduced`).
4. **Enumerate variable orders.** For every total order on the variables
   (`all_orders`), check whether the system is *increasing* for that order
   (`all_non_increasing`). For each divisor `f` in primitive form, the check
   computes a finite basis of the `Z`-module generated by `f` using a
   fix-point iteration (`basis_of_divmodule`), intersects it with the module
   of polynomials on variables at most the leading variable of `f`, and
   compares the Hermite normal form of that intersection with the HNF of
   `⟨f⟩`. Any basis vector appearing in the former but not the latter is a
   witness of non-increasingness.
5. **Split on non-increasingness witnesses.** For each witness `(f, G)`, and
   each `g ∈ G`, the algorithm adds a new equality `c·f − g = 0` for every
   integer `c` in `[−S, S]` (with `S = Σ|g_i|`), taking the cross-product of
   these choices to form new sub-systems (`all_disj_from_noninc`). Every such
   equality strictly reduces the number of free variables, which is the key
   to termination. Each new sub-system is pushed back onto the worklist and
   re-enters step 2.
6. **Collect results.** Sub-systems that reach "increasing" status are
   returned as `OrdLinDivs` objects pairing the system with the order that
   witnesses increasingness.

Two flags tune the amount of work done:

- `check_sym_inc` (default `True`): stop at the first order that witnesses
  symbolic increasingness, instead of enumerating all orders.
- `use_all_cx_inc` (default `True`): when splitting on non-increasingness,
  use every counterexample; setting it to `False` keeps only the first.

### Example

Consider the example from the cited SODA paper, also encoded in
`tests/test_norm.py::test_soda_paper_example`:

```
x + 1 | y − 2   ∧   x + 1 | x + y
```

In the tuple representation used here (each tuple is
`(coef of x, coef of y, constant)`), this is

```python
LinDivs(divisors =((1, 0, 1), (1, 0, 1)),
        dividends=((0, 1, -2), (1, 1, 0)))
```

Running `lds.norm()` on this system proceeds roughly as follows. All divisor
LHS coefficients are already non-negative, so the left-positivity step is a
no-op. There are no equations or inequalities, so elimination is a no-op and
gcd-reduction leaves the system unchanged. When checking increasingness for
the order `x < y`, the primitive divisor `x + 1` is flagged as non-increasing:
the module intersection contains `x` and the constant `1`, but `⟨x + 1⟩`
does not, so the HNF comparison produces a witness. The algorithm then
splits disjunctively, adding equalities of the form `c·(x + 1) − 1 = 0` and
`c·(x + 1) − x = 0` for small integers `c`. Each resulting sub-system has
one fewer free variable and is re-normalized recursively, ultimately yielding
`OrdLinDivs` results on every branch. (This particular example is
unsatisfiable over the naturals — normalization still terminates with the
disjunctive case split, which is what the test asserts.)

## Requirements to implement normalization
Given a system of divisibilities and linear constraints, we can implement
Lipshitz' transformation into increasing normal form if we have:
- [X] A way of obtaining a (hybrid) semilinear representation of the solutions
  of a system of linear equations---which can be used to update the system of
  divisibilities via an affine change of variables while not increasing the
  number of variables
- [X] A way of enumerating the sign of the left-hand sides of divisibility
  constraints, towards obtaining left-hand sides with all positive
  coefficients, and adding them as inequalities to then remove them via the
  method described above
- [X] A way of obtaining a finite representation of the module of the
  primitive polynomials of the left-hand sides. In short, (see Section C.1
  from the extended version of the cited SODA paper) we get a finite spanning
  set of the module.
- [X] A way of checking whether a given system is increasing for a given
  (semantic) order over the variables, based on the representation above. For
  this, we can compare the Hermite normal form (HNF) of matrices spanning (1) the
  Z-module generated by the primitive form and (2) the intersection of the
  modules of the primitive polynomials and the relevant set of polynomials.
  (For an algorithm to compute the intersection of modules, see Chapter 2 from
  H. Cohen's __A Course in Computational Algebraic Number Theory__. Flint
  implements HNF algorithms based on LLL.)
- [X] An algorithm to deal with a witness of nonincreasingness to split
  (disjuntively) into subformulas with less variables (**Key for termination:
  each subformulas has to have strictly less variables**)
- [X] A recursive-style algorithm to keep finding nonincreasing subformulas to
  treat with the procedure above.

# Miscellaneous

## Design choices
- We are only encoding systems, so no disjunctions.
- Variables are quantified over the naturals, not the integers. 
- Everything is tuples (of tuples).
- We don't encode Ax = b but rather Ax - b = 0 and the last "coefficient" is
  the constant in the polynomial. Similar weirdness applies to inequalities.

## Dependencies
For now, Z3, pyyaml and flint
